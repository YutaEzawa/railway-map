# split-lines — 区間分割の自動化

`data/lines.csv` の `状態` が空（pending）の路線を選択し、Yahoo!路線情報の時刻表から
区間ごとの本数を取得して、必要に応じて区間分割を自動生成するスキル。

## 2つの実行モード

| モード | いつ使う | 進め方 |
|--------|----------|--------|
| **逐次モード** | 数路線だけ・1路線を個別に・動作確認 | 下の「逐次モードの手順」（ステップ1〜4）を直接実行 |
| **並列バッチモード** | 全国拡張・pending を大量にまとめて処理 | 下の「並列バッチモード」。専用エージェントを並列スポーンして高速化 |

「全国」「一括」「まとめて」「pending を全部」「N路線を並列で」などと言われたら **並列バッチモード**。
特に指定がなく少数なら逐次モード。

## 起動フレーズ（依頼の文言の解釈ルール）

ユーザーの文言から **モード・路線数・並列度** を次の規則で確定する。曖昧なら確認せず既定値で進める。

| ユーザーの文言に含まれる語 | モード | 補足 |
|---|---|---|
| 「逐次」「ひとつずつ」「順番に」「確認しながら」 | 逐次モード | |
| 「並列」「バッチ」「全国」「一括」「まとめて」「pending を全部」 | 並列バッチモード | |
| モード語が無く 路線数 ≤ 5 | 逐次モード（既定） | |
| モード語が無く 路線数 > 5 または「全部」 | 並列バッチモード（既定） | |

| 数値の指定 | 解釈 |
|---|---|
| 「N路線」「N本」「N件」 | 処理する pending 路線数（逐次は `--count N`／並列は pending を N 件に絞ってバッチ分割） |
| 「並列度K」「Kバッチ」「Kエージェント」「K並列」 | 並列スポーン数。**未指定なら 3**。礼儀のため最大 3 に丸める |
| 「全部」「すべて」「pending 全件」 | pending 全件（現状約100）を対象 |
| 路線名（例「南武線 横浜線」） | その路線のみ（`--line`）。複数なら並列バッチの対象集合になる |

未指定時の既定: **並列度 3 / 確認あり（ドライランをユーザーに見せてから反映）**。
「確認なし」「反映まで」「そのまま書き込んで」と言われたらドライラン提示を省いて `--write` まで進む。

## 並列バッチモード（全国拡張向け）

### 全体方針: 並列スクレイプ → 直列反映

律速は Yahoo! スクレイプ（1.2秒間隔・キャッシュ済みは高速）。ここで効く性質：

- **ドライラン（`--write` なし）は CSV を一切変更せず `.data/yahoo-cache/` を温めるだけ。**
- いったんキャッシュが温まれば、後続の `--write` はキャッシュヒットで高速。

よって「並列でキャッシュを温め → 1プロセスで直列に書き込む」と、速度・CSV安全・礼儀を両立できる。

```
pending 抽出
  ↓ N(=3) バッチに分割
┌─────────── 並列（background, ドライランのみ）───────────┐
│ train-count-fetcher  train-count-fetcher  train-count-fetcher │  ← .data/yahoo-cache/ を温める
│   バッチA              バッチB              バッチC            │     CSV は触らない＝競合なし
└────────────────────────────────────────────────────────┘
  ↓ 全完了（自動通知を待つ）
1プロセスで --write（キャッシュ済みで高速・単一ライター＝競合なし）
  ↓
build-railways.cjs を1回
```

### 手順

#### ステップ A: pending 路線の抽出とバッチ分割

```bash
awk -F, 'NR>1 && $5=="" {print $1}' data/lines.csv | sort -u
```

得られた路線を **排他的な N バッチ** に分ける。
- **並列度 N は 2〜3 を上限の目安にする**（Yahoo! への礼儀。1.2秒間隔はプロセス単位なので
  N 並列は実効 N 倍の負荷になる。むやみに増やさない）。
- バッチは均等な件数で機械的に分ければよい（地方・エリアでまとめると進捗管理しやすいが必須ではない）。

#### ステップ B: 専用エージェントを並列スポーン（ドライランのみ）

各バッチを `train-count-fetcher` エージェント（`.claude/agents/train-count-fetcher.md`、Haiku）に
**同一メッセージ内で `run_in_background: true`** で渡す。プロンプトに必ず明記すること：

- 担当路線（バッチの路線名リスト）
- **「ドライランのみ。`--write` は実行しない（親が直列で行う）」**
- 結果（auto_v1 / auto_v2 / auto_fail・本数・区間）を表で報告すること

```
Agent(subagent_type:"train-count-fetcher", run_in_background:true,
      prompt:"次の路線をドライランで調査。--write は実行しない。結果を表で報告: <バッチAの路線>")
Agent(subagent_type:"train-count-fetcher", run_in_background:true,
      prompt:"... <バッチBの路線>")
Agent(subagent_type:"train-count-fetcher", run_in_background:true,
      prompt:"... <バッチCの路線>")
```

全エージェントの完了通知を待つ。各エージェントの報告（特に auto_fail とその理由）を集約する。

#### ステップ C: 直列で CSV に反映（キャッシュ済みで高速）

キャッシュが温まっているので、**1プロセスで一括 `--write`** すればよい（単一ライター＝競合なし）。
**処理は止めない**：問題なしは auto_v1/auto_v2、要確認は auto_review、失敗は auto_fail として
すべて書き込み、要確認・失敗は `data/review.md` に自動集約される。

```bash
node split-lines/split-lines.cjs --count <pending総数以上の値> --write
```

- 反映後、差分（`git diff data/lines.csv`）を確認してユーザーに要約を見せる。
- **確認はまとめて**: `data/review.md`（auto_review / auto_fail のチェックリスト）を最後に提示し、
  ユーザーが1ファイルで確認できるようにする。直したら該当行を `manual` にして `--review` で再生成。

#### ステップ D: GeoJSON を再生成（地図反映に必須・1回だけ）

```bash
node scripts/build-railways.cjs
```

`.data/` が無い環境では CSV は更新済みだが GeoJSON は未反映である旨を伝える。

> 並列で `build-railways.cjs` を走らせない。必ずステップ D で一括 1 回だけ。

## 状態値の意味

| 状態       | 意味                                           |
|------------|------------------------------------------------|
| （空）     | 未処理（pending）。このスキルの処理対象         |
| `manual`   | 手動で駅列・区間を記載済み。変更しない          |
| `auto_v1`  | 自動処理済み（均一本数、区間分割なし）          |
| `auto_v2`  | 自動処理済み（区間分割あり、駅列を自動生成）    |
| `auto_review`| 自動処理したが**要確認**（中間V字凹み等）。値は暫定で反映済み。`data/review.md` に集約 |
| `auto_fail`| 自動処理に失敗（Yahoo!解決不可・駅順序ずれ等）。`data/review.md` に集約 |

> **確認はまとめて行う**: `auto_review` と `auto_fail` は処理を止めず、`data/review.md`
> （チェックリスト）に自動集約される。`--write` のたびに更新され、`node split-lines/split-lines.cjs --review`
> でも CSV から再生成できる。確認して直したら該当行の `状態` を `manual` にすればリストから外れる。

## 逐次モードの手順

少数の路線を確認しながら処理する場合はこちら。

### ステップ 1: 対象路線の選択

`状態` が空の行から `--count N`（デフォルト5）路線分を選択してドライランを実行する。

```bash
node split-lines/split-lines.cjs [--count N] [--line 路線名] [--fresh]
```

### ステップ 2: 結果の確認

スクリプト出力のサマリーをユーザーに見せる。ポイント：
- **均一（auto_v1）**: 本数に問題ないか確認する
- **区間分割（auto_v2）**: 生成された区間・本数・駅列を確認する
- **失敗（auto_fail）/ ⚠ 要確認（auto_review）**: 後述「自己検証」。**処理は止めず**に `data/review.md`
  へ集約されるので、バッチ全体が終わってから**まとめて確認**する

#### 自己検証（確認はまとめて）

スクリプトは次の2つを自動チェックする。検出しても処理は止めず、該当路線を `data/review.md` に集約する。
**最後にこのファイルをまとめて確認**すればよい（1件ずつ止まらない）。

| 検出 | 出力 | 状態 | 意味 |
|------|------|------|------|
| **連結ギャップ（方式D）** | `✗ … 大ギャップ NNkm` | `auto_fail` | セグメント連結が NNkm ワープ＝駅順序が信頼できない。本数は既定値のまま |
| **本数の中間V字凹み** | `⚠ 要確認: ○○〜△△ が N本…` | `auto_review` | 本数は折返し駅でしか減らないので、線途中の極小は駅順序ずれ/カウント誤差の兆候。値は暫定で反映済み |

確認後の直し方: 実態を調べ、正しい区間・本数にして該当行の `状態` を `manual` にする → `data/review.md` から外れる。

> 例: 高崎線は当初 `118 → 96 → 118 → 64` と中間が凹む警告が出た（auto_review）。実態は
> 「大宮〜籠原 ≈118／籠原〜神保原 ≈64」の2区間（籠原が折返し駅）で、手動 `manual` で確定した。

### ステップ 3: CSV への反映

ユーザーが OK なら `--write` を付けて実行する。

```bash
node split-lines/split-lines.cjs [--count N] [--line 路線名] [--write]
```

### ステップ 4: GeoJSON の再生成（地図反映に必須）

```bash
node scripts/build-railways.cjs
```

`.data/` が無い環境では CSV は更新済みだが GeoJSON は未反映である旨を伝える。

## 対応オプション（ユーザーが自然言語で指定できる）

| ユーザーの指示               | スクリプトへの変換                        |
|------------------------------|-------------------------------------------|
| 「5路線分」「10本ずつ」など  | `--count N` 引数に変換                    |
| 「京急本線だけ」など路線名   | `--line 路線名` 引数に変換                |
| 「キャッシュを使わず」       | `--fresh` フラグを追加                    |
| 「確認なしで更新して」       | ドライラン確認を省いてステップ 3 に進む   |
| 「確認リストを更新/見せて」  | `--review`（CSV から `data/review.md` を再生成・読み取り専用）|

## 処理アルゴリズム

1. `public/data/railways.geojson` と `public/data/stations.geojson` から、路線上の駅を
   路線ジオメトリへの射影によって順序付き（始端→終端）で取得する。
   - セグメント連結は **真の終端（出現1回の端点）から開始**して途中開始によるワープを防ぐ（方式A）。
   - 連結時の最大ギャップが大きい（既定 5.5km 超）と **駅順序が信頼できない → auto_fail**（方式D）。
2. 始端駅・終端付近の駅でYahoo!時刻表をスクレイプし、平日・片方向の本数を数える。
3. 本数が一様（差が10%以内 or 5本以内）→ **auto_v1**（本数を確認・更新）
4. 本数が異なる → 二分探索で変化点を特定 → **auto_v2**（区間ごとの行を生成）
   - 区間本数に **中間V字凹み（内部の極小）** があれば `⚠ 要確認` を出す（折返し駅でしか
     減らないはずなので、駅順序ずれ/カウント誤差の自己検知）。
5. 解決不可（駅IDが不明、路線が見つからない等）→ **auto_fail**（理由を記録）

## トラブルシューティング

| 症状 | 対処 |
|------|------|
| `始端「○○」で本数取得失敗` | `split-lines.cjs` の `LINE_NAME_TO_YAHOO` に路線名対応を追加 |
| GeoJSONに駅データなし | `public/data/stations.geojson` に路線名が含まれているか確認（corridor系は別名の場合がある） |
| 方向が一意に決まらない | `split-lines.cjs` の `SECTION_OVERRIDES`（未実装の場合は `TOWARD_OVERRIDES`）に路線・向き先を追加 |
| 変化点がおかしい | ドライランで区間・本数を確認し、`--line` で個別に再実行 |
| スクレイプ結果が古い | `--fresh` でキャッシュを無効化 |

## 種別ごとの停車本数（駅サイズ＋クリックで種別内訳）

区間分割（線の太さ＝通過本数）とは**別チャンネル**で、**駅の丸サイズ＝停車本数**、
**駅クリックで種別ごと・合計の停車本数**を表示する機能。急行/各停で停車駅が違う路線
（西武池袋線・京王線など）で「1路線に1本数」の無理を避け、駅が各自の数字を持つ。

- 線の太さ＝通過本数（`data/lines.csv` の `本数`。区間分割スキルの担当）
- 駅サイズ＝停車本数（`stopTrains`）／クリックの種別内訳（`stopTypes`）＝**この節の担当**

対象は区間分割（`data/lines.csv`）とは独立で、`data/lines.csv` は変更しない。

### いつ使う（起動フレーズ）

「停車本数を駅に表示」「種別ごとの停車本数」「駅サイズに反映」「この路線も急行/各停の差を出す」
「◯◯線を駅サイズ対応に」などと言われたとき。**区間分割の依頼とは別物**（混同しない）。

### 手順

#### ステップ 1: 対象路線を `TARGET_LINES` に追加

`train-counts/scrape-station-stops.cjs` 先頭の `TARGET_LINES` に `{ line, terminals }` を足す。

- `line` — `public/data/stations.geojson` の路線名（駅一覧はここから自動取得）。
- `terminals` — **上り（東京寄り終端）方面ラベルの優先順**の配列。各駅で、この方面へ向かう
  向きの平日発車本数＝停車本数を数える。先頭が見つからなければ次を試す
  （例: 西武池袋線は飯能以西の秩父線区間で上りラベルが「飯能」になるため `['池袋','飯能']`）。
- 難所の駅だけ次の任意フィールドで上書き：
  - `stationLine: { 駅名: 別路線名 }` — その駅で数える路線を差し替え（例: 初台・幡ヶ谷は
    京王線に出ないので `京王新線`）。
  - `stationToward: { 駅名: [向き先] }` — その駅だけ方面ラベルを上書き（例: 京王新線の上りは
    都営新宿線直通で終点が本八幡になるので `['本八幡']`）。

#### ステップ 2: 停車本数をスクレイプ

```bash
node train-counts/scrape-station-stops.cjs            # TARGET_LINES 全路線
node train-counts/scrape-station-stops.cjs 京王線       # 路線名で絞り込み（部分一致）
FRESH=1 node train-counts/scrape-station-stops.cjs ... # キャッシュ無視で再取得
```

- 出力は `train-counts/station-stops.json`（`{ 路線: { 駅名: { total, types } } }`）。
- 種別名は `特急◯◯N号` の愛称・号数を落として `特急` に正規化（準特急・通勤特急は残す）。
- **⚠ 路線名で絞ると JSON はその路線だけで上書きされる。** 既存路線ぶんを残すなら
  **引数なし**で全 `TARGET_LINES` を回すこと（絞り込みは新規追加路線の動作確認用）。
- `✗ … 未解決` が出たら理由（駅ID未解決／方向不定／路線なし）を見て、ステップ1の
  `stationLine`/`stationToward`、または `scrape-yahoo-trains.cjs` の
  `STATION_ID_OVERRIDES`（駅名→駅ID）を追加して再実行。

#### ステップ 3: GeoJSON を再生成（地図反映に必須）

```bash
node scripts/build-railways.cjs
```

`build-railways.cjs` が `station-stops.json` を読み、駅フィーチャに `stopTrains`（合計・
駅サイズ用）と `stopTypes`（種別内訳の JSON 文字列・ポップアップ用）を付与する
（`stopTrainsFor()`。乗換駅で複数路線にデータがあれば合計の多い路線を採用）。
`src/map/MapView.tsx` は `stopTrains` で `circle-radius` を連続変化させ、駅クリックで
`stopTypes` を本数降順に表示する（データの無い駅は従来の `major` 2値サイズ・駅名のみ）。

### 近似・注意

- 直通サービス（西武→副都心線・京王→都営新宿線 等）は Yahoo 上で別路線扱いのため
  停車本数に含まれない。急行/各停の停車パターン差を見るには十分な近似。
- 数え方は片方向（上り）。線の太さ側（`本数`）と同じく平日・片方向で揃えている。
- robots.txt 遵守・キャッシュ・駅ID解決は区間分割と共通の Yahoo スクレイパ基盤
  （`train-counts/scrape-yahoo-trains.cjs` のヘルパー）を再利用している。

## robots.txt の遵守

Yahoo!路線情報の robots.txt を厳守している。1.2 秒間隔・許可パスのみ取得・
`.data/yahoo-cache/` にキャッシュ（gitignore 済み）。

## 設定ファイル

`split-lines/split-lines.cjs` の先頭定数を編集する。

- `LINE_NAME_TO_YAHOO` — CSV の路線名 → Yahoo 表示名の対応（食い違いがある路線のみ）
- `isSameFreq()` — 「均一」とみなす閾値（デフォルト: ±10% or ±5本）
